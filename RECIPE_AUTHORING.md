# Authoring Recipes

This guide explains how recipes are structured in Arconia Migrations and how to add or update them. It assumes familiarity with OpenRewrite's own [Authoring Recipes](https://docs.openrewrite.org/authoring-recipes) documentation and focuses on the project-specific conventions and tooling on top of it. It complements the conventions captured in the ADRs:

* [ADR 1: Recipe naming](adr/0001-recipe-naming.md): identifiers, `Upgrade`/`Migrate` patterns, acronym capitalization, required `displayName`/`description`.
* [ADR 2: Deprecating and removing recipes](adr/0002-deprecating-and-removing-recipes.md): how to rename or retire a recipe without breaking users.

Read both before adding or changing a recipe. The [Versioning and Support](https://docs.arconia.io/arconia-migrations/latest/versioning.html) policy explains why recipe identifiers are part of the public API.

## Module and resource layout

Recipes are organized into modules by area (for example `rewrite-spring` and `rewrite-testing`). Within each module:

* **Declarative recipes** are YAML under `src/main/resources/META-INF/rewrite/<area>/...`. Prefer these: most migrations are a composition of existing building blocks.
* **Imperative recipes** are Java under `src/main/java/io/arconia/rewrite/<area>/...`, used when a transformation needs logic that declarative YAML can't express. Every package is `@NullMarked` (null-safety is enforced on `src/main`; see the `code-quality-conventions` build plugin).

Property migrations follow a three-layer composition, from mechanical to bespoke:

```
…PropertiesGenerated_X_Y   (generated YAML, never hand-edited)
        ▲ composed first by
…Properties_X_Y            (hand-written wrapper: adds what metadata can't express)
        ▲ composed by
UpgradeSpring<Area>_X_Y    (the version aggregate users run)
```

## Generating property-migration recipes

Spring publishes deprecation information in each module's `META-INF/spring-configuration-metadata.json`. Three Gradle tasks turn that metadata into a generated recipe, one per ecosystem:

```shell
./gradlew generateSpringAiProperties    --args='2.0.0'      # spring-ai-bom version
./gradlew generateSpringBootProperties  --args='4.1.0'      # spring-boot-dependencies version
./gradlew generateSpringCloudProperties --args='2025.0.0'   # spring-cloud-dependencies version
```

Each task runs a `*PropertiesMigrationGenerator` entry point that supplies a `Spec` (BOM coordinates, module prefix, output path, recipe-name pattern) to the shared `PropertiesMigrationGenerator`. The generator:

1. Resolves the BOM for the given version, walking parent POMs and `<scope>import</scope>` BOM imports.
2. Selects the modules whose artifact id starts with the configured prefix (for example `spring-ai-autoconfigure-`).
3. Downloads each module jar (cached under `.generated/`) and reads its `spring-configuration-metadata.json`.
4. Emits one entry per deprecated property: a `RenameSpringBootPropertyKey` when the metadata carries a `replacement`, otherwise a `CommentSpringBootProperty` (for removed or unbindable properties, including the deprecation reason when present).
5. Writes a `…-properties-generated.yml` file named `UpgradeSpring<Area>PropertiesGenerated_X_Y`, with a header marking it as generated.

The generated file starts with:

```yaml
# This file is generated by SpringAiPropertiesMigrationGenerator.
# Do not edit manually. Update the upstream configuration metadata.
```

**Never hand-edit a generated file.** To refresh it, re-run the task for a new version; to change *what* is generated, change the generator (see below).

### What the generator deliberately drops

The generator omits two classes of entry and prints them to the console so you can handle them in
the wrapper:

* **Conflicting keys**: the same old key mapped to different targets across modules. Letting the first match win would be wrong for a user on a different module, so these are dropped.
* **Redundant parent renames**: a parent-key rename already fully covered by leaf renames; keeping it would sweep undocumented subkeys.

These are *data-shape* invariants, so the detection lives in the generator (the `find…` helpers), not in a per-version workaround. Add new structural rules there; keep one-off, version-specific handling in the wrapper.

## Wrapping generated recipes

The hand-written wrapper (`…-properties.yml`, named `UpgradeSpring<Area>Properties_X_Y`) composes the generated recipe **first**, then adds everything the configuration metadata can't express:

```yaml
recipeList:
  - io.arconia.rewrite.spring.ai1.UpgradeSpringAiPropertiesGenerated_1_0   # generated, first
  - io.arconia.rewrite.spring.boot.properties.RenameSpringBootPropertyKey: # manual additions
      oldPropertyKey: "spring.ai.chat.memory.jdbc.initialize-schema"
      newPropertyKey: "spring.ai.chat.memory.repository.jdbc.initialize-schema"
  - io.arconia.rewrite.spring.boot.properties.ChangeSpringBootPropertyValue:
      propertyKey: "spring.ai.chat.memory.repository.jdbc.initialize-schema"
      oldPropertyValue: "true"
      newPropertyValue: "always"
  - io.arconia.rewrite.spring.boot.properties.DeleteSpringBootProperty:
      propertyKey: "spring.ai.openai.chat.enabled"
```

Put these in the wrapper:

* Renames that change nesting depth (the metadata only records flat key replacements).
* Value changes (`ChangeSpringBootPropertyValue`), e.g., a boolean becoming an enum.
* Deletions (`DeleteSpringBootProperty`).
* The conflicting keys the generator dropped (often behind a precondition that picks the right target for the user's actual modules).
* Anything else not derivable from `spring-configuration-metadata.json`.

The wrapper is then composed into the version aggregate recipe (for example `UpgradeSpringAi_1_0`). Because bespoke logic lives only in the wrapper, regenerating the generated file is always safe.

## Parser classpath and TypeTables

Recipes and tests that rely on type attribution (Java recipes using `JavaTemplate`, `UsesType`, or
`MethodMatcher`, and Java test fixtures) need the relevant types on the parser classpath. Declare
them in the `recipeDependencies` block of the module's `build.gradle`:

```groovy
recipeDependencies {
    parserClasspath "org.springframework:spring-core:7.0.+"        // needed at recipe runtime
    testParserClasspath "org.springframework:spring-test:7.0.+"    // needed only by test fixtures
}
```

* **`parserClasspath`**: types referenced by main-source recipes via `classpathFromResources(...)`. These ship in the recipe's runtime TypeTable.
* **`testParserClasspath`**: types needed only to attribute test fixtures; not shipped.

Prefer floating versions (`7.0.+`) so the entries don't need bumping on every upstream patch.

The build resolves these into committed TypeTables (`classpath.tsv.gz`) rather than downloading at
test time. After changing `recipeDependencies`, regenerate and commit them:

```shell
./gradlew :rewrite-spring:createTypeTable       # writes src/main/resources/META-INF/rewrite/classpath.tsv.gz
./gradlew :rewrite-spring:createTestTypeTable   # writes src/test/resources/META-INF/rewrite/classpath.tsv.gz
```

For a recipe that migrates from one library version to another, the test parser must resolve the _before_ API while a `JavaTemplate` resolves against the _after_ API. Keep those on the correct classpath (before → parser/`testParserClasspath`, after → the recipe's `classpathFromResources`) rather than mixing both versions of the same artifact into one classpath.

## Testing recipes

Tests implement OpenRewrite's `RewriteTest`. Set the recipe under test once in `defaults`, then assert before/after with the source-spec helpers:

```java
public class UpgradeSpringAiProperties_1_0_Tests implements RewriteTest {

    @Override
    public void defaults(RecipeSpec spec) {
        // declarative recipe by id; use spec.recipe(new MyRecipe(...)) for an imperative one
        spec.recipeFromResources("io.arconia.rewrite.spring.ai1.UpgradeSpringAiProperties_1_0");
    }

    @Test
    @DocumentExample // the canonical before/after shown in docs
    void updateProperties() {
        rewriteRun(
                //language=properties
                properties(
                        """
                        spring.ai.chat.memory.jdbc.initialize-schema=false
                        """,
                        """
                        spring.ai.chat.memory.repository.jdbc.initialize-schema=never
                        """,
                        s -> s.path("src/main/resources/application.properties"))
        );
    }
}
```

Guidelines:

* Use the helper that matches the source type: `properties(...)`, `yaml(...)`, `java(...)`, `pomXml(...)`, `buildGradle(...)`.
* Omit dependency versions in `buildGradle` fixtures: the Gradle tooling model tolerates version-less dependencies, and pinning a version makes the test break on the next upstream patch.
* Mark one representative case with `@DocumentExample`.
* Group test cases by conceptual migration (one test per release-note bullet / PR theme), not one per individual recipe entry.

Run a module's tests with `./gradlew :rewrite-spring:test`, or the whole build with `./gradlew build`.

## Recipe catalog documentation

`docs/modules/ROOT/pages/recipe-catalog/` holds an auto-generated catalog of every published recipe, grouped by module and category, with composite recipes listed before the leaf recipes they compose. It is built from the OpenRewrite recipe model (display names, descriptions, options, composition tree) plus the before/after example from each recipe's `@DocumentExample` test, rather than hand-written, and the build-only `docs-generator` module owns it.

After you add, rename, remove, or change the metadata of a recipe, regenerate and commit the pages:

```shell
# Rewrites docs/.../recipe-catalog/ and the nav.adoc recipe-catalog block
./gradlew :docs-generator:generateRecipeCatalogReference
```

**Never hand-edit the catalog pages** or the `// tag::recipe-catalog[]` region of `docs/modules/ROOT/nav.adoc`; both are overwritten on regeneration. `./gradlew build` runs `:docs-generator:validateRecipeCatalogReference`, which fails if the committed pages drift from the recipes, so CI catches a forgotten regeneration.
